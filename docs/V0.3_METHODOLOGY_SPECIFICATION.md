# V0.3 Methodology Specification

## 1. Purpose

Version 0.3 replaces a universal header checklist with a response-profile model.
The same missing header can be material for an authenticated HTML application,
contextual for an API, and irrelevant for a response that does not create a browser
document. The objective is to reduce misleading penalties while keeping the
assessment transparent and reproducible.

This document is normative for methodology version `0.3.0`.

## 2. Profile Taxonomy

### `app` - Authenticated Web Application

A browser-rendered application that handles user state, protected workflows, or
sensitive data. It receives the broadest browser-security baseline.

Evidence that can support this classification:

- `Content-Type: text/html`;
- session state through `Set-Cookie`;
- an authentication challenge through `WWW-Authenticate`;
- a `private` or `no-store` cache policy.

Auto-detection requires HTML plus at least two application signals. This threshold
is intentionally conservative because headers cannot prove that a route is
authenticated or sensitive.

### `api` - API Response

A machine-readable REST, GraphQL, or XML response that does not create a browser
document. Auto-detection requires an explicit JSON or XML media type, including
structured suffixes such as `+json` and `+xml`.

Browser document controls are not scored merely because the endpoint can be reached
with a browser. Transport and authoritative media typing remain scored.

### `brochure` - Public Brochure Site

Public HTML content with limited authenticated state. It prioritizes transport,
content, MIME, framing, referrer, and browser-feature controls without assuming
application-only process isolation.

Plain HTML without enough application evidence maps to this profile. Missing or
ambiguous `Content-Type` also falls back to this profile with `low` confidence.

## 3. Profile Resolution

Resolution follows this order:

1. An explicit `--profile app|api|brochure` selection always wins.
2. A recognized machine-readable media type selects `api` with `high` confidence.
3. HTML plus at least two application signals selects `app` with `medium` confidence.
4. Other HTML selects `brochure` with `medium` confidence.
5. An ambiguous response selects `brochure` with `low` confidence.

Every result records the requested profile, selected profile, confidence, and
decision evidence. Auto-detection is a usability aid, not an application classifier.

## 4. Profile Weights

Each profile totals 100 points. A zero weight means the rule is contextual or not
applicable, not that the header has no security value in every deployment.

| Header | `app` | `api` | `brochure` |
| --- | ---: | ---: | ---: |
| `Strict-Transport-Security` | 20 | 60 | 30 |
| `Content-Security-Policy` | 25 | 0 | 30 |
| `X-Content-Type-Options` | 10 | 40 | 15 |
| `X-Frame-Options` or CSP `frame-ancestors` | 5 | 0 | 10 |
| `Referrer-Policy` | 10 | 0 | 10 |
| `Permissions-Policy` | 10 | 0 | 5 |
| `Cross-Origin-Opener-Policy` | 10 | 0 | 0 |
| `Cross-Origin-Resource-Policy` | 10 | 0 | 0 |
| **Total** | **100** | **100** | **100** |

### Weighting Rationale

- `app` distributes weight across transport, executable content, browser capabilities,
  and origin isolation because all can affect authenticated browser workflows.
- `api` concentrates weight on transport and MIME authority. OWASP REST guidance
  notes that several browser document headers provide little or no benefit on
  non-HTML API responses.
- `brochure` prioritizes delivery and content controls while avoiding universal
  penalties for advanced process-isolation headers.

The weights are an explicit project risk model. They are not copied from Mozilla
HTTP Observatory, OWASP, NIST, or a regulatory scoring scheme.

## 5. Finding Semantics

| Status | Meaning |
| --- | --- |
| `pass` | The observed value satisfies the implemented profile check. |
| `warning` | The header is present but weak, incomplete, or context-sensitive. |
| `missing` | A scored profile expectation was not observed. |
| `info` | A non-scored observation is present or absent. |
| `review` | A contextual value requires operator review. |
| `not_applicable` | The response profile does not create the relevant security context. |
| `observed` | A disclosure-related header was present; this is not proof of vulnerability. |

Partial credit is explicit in each evaluator. For example, an HSTS `max-age`
below one year receives less credit than the ASVS-aligned one-year baseline.
A CSP that permits unsafe script sources receives partial credit and a high-severity
warning. An explicit CSP `frame-ancestors` directive satisfies the framing control
even when the legacy `X-Frame-Options` header is absent.

## 6. Citation Policy

Every rule links to one or more of:

- a primary web or Internet specification;
- an official security standard or guidance source;
- peer-reviewed research where deployment weakness matters;
- a platform reference for implementation details.

Identifiers are version-pinned where versions exist. OWASP ASVS references use
v5.0.0 identifiers. NIST SP 800-53 Rev. 5 relationships are labelled
`control-informed`; the tool does not test control implementation or compliance.

The complete source inventory is in [CITATION_MANIFEST.md](CITATION_MANIFEST.md).

## 7. Non-Goals

Version 0.3 does not:

- prove that an application is secure or compromised;
- certify OWASP ASVS, NIST, GDPR, or other regulatory compliance;
- crawl a site, enumerate routes, authenticate, fuzz, exploit, or bypass controls;
- inspect application bodies, JavaScript behavior, cookies, TLS configuration, or backend code;
- validate CSP against the application's complete resource graph;
- infer data sensitivity, business purpose, or authorization from one response;
- replace threat modelling, code review, vulnerability testing, or professional assessment;
- claim equivalence with Mozilla HTTP Observatory or OWASP projects.

By default, the request remains inside the original origin. Same-host HTTP-to-HTTPS
upgrades are allowed; other origin changes require the explicit
`--allow-cross-origin-redirects` option and operator authorization.

## 8. Reproducibility And Change Control

Regression tests use local JSON fixtures and mocked request outcomes for profile,
request-policy, and rendering behavior. They do not depend on remote sites. Any
change to profile taxonomy, weight,
applicability, partial credit, or citation mapping requires:

1. a methodology version decision;
2. updated deterministic fixtures and tests;
3. updated documentation and release notes;
4. successful local and CI test matrices;
5. browser QA for HTML changes.
