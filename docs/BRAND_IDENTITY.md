# Response Ledger Identity

## Product relationship

**Response Ledger** is the public product identity for **Security Headers
Auditor**, created by **Vasileios Tsalikidis**. It gives the project a concise,
memorable presentation without changing its stable technical identifiers:

- repository: `security-headers-auditor`;
- Python package and CLI: `security-headers-auditor`;
- releases, schemas, and evidence: Security Headers Auditor.

The product name therefore reads **Response Ledger / Security Headers Auditor**
where a reader needs both the public identity and the implementation context.

## Primary expression

**Response Ledger**

*Response context. Reviewable evidence.*

The name describes the project’s actual value: it evaluates a deliberately
bounded HTTP response and retains a reviewable record for change-control work.
It does not describe a generic scanner, a security certification, or a claim to
prove an endpoint secure.

## Visual direction

The primary mark is a wordmark, not an icon. It uses warm ivory space,
graphite for **Response**, deep teal for **Ledger**, and a fine copper rule. The
visual language is editorial and technical: calm hierarchy, useful whitespace,
and visible evidence rather than stereotypical “cyber” decoration.

| Token | Value | Purpose |
| --- | --- | --- |
| Paper | `#fffefb` | Primary background |
| Graphite | `#202426` | Reading and “Response” |
| Deep teal | `#0d5c63` | “Ledger” and active identity accent |
| Copper | `#a96a35` | Fine rule and sparing secondary accent |
| Quiet surface | `#f6f5f0` | Secondary panels |

Avoid shields, locks, circuit traces, terminals, grids, neon, gradients,
glowing effects, and decorative icon fields. They do not communicate the
project’s actual distinction and would make it visually interchangeable with
generic cybersecurity or AI tooling.

## Evidence-backed language

Use these phrases only in their supported sense:

- **Local-first and read-only:** targets are operator-supplied; the tool has no
  hosted target collection, telemetry, accounts, crawling, authentication, or
  active probing.
- **Context-aware:** `app`, `api`, and `brochure` profiles change applicability
  and scoring; detection is conservative and can be overridden explicitly.
- **Reviewable evidence:** reports, policy thresholds, approved baselines,
  route comparisons, and compact evidence capsules support recorded review over
  time.

Do not describe a score as a vulnerability count, a compliance decision, or
proof that a target is secure. The complete intended-use boundary remains in
[Responsible Use](RESPONSIBLE_USE.md).
