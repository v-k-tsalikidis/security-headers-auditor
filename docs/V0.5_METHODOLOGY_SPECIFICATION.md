# v0.5 Methodology Specification

## Status

This document defines the implemented v0.5 workspace interface and evidence
taxonomy. The v0.4 audit, profile, assurance, and regression semantics remain
the authoritative engine and must not be reimplemented in the frontend. Tool
and methodology versions are intentionally independent; see
[`ADR 0004`](adr/0004-tool-and-methodology-versioning.md).

The implementation remains subject to the evidence in the v0.5 release gate;
this document is not a release-complete claim.

## Product Boundary

Version 0.5 adds a local workspace interface for configuring and running the
existing Security Headers Auditor. It does not add a new scanner category.

The workspace supports:

- explicit authorized target inventory;
- response-profile and assurance expectation configuration;
- one-at-a-time and policy assurance runs through the existing Python engine;
- result inspection;
- candidate and approved baseline review;
- minimal local persistence;
- explicit JSON, Markdown, HTML, SARIF, and JUnit export where applicable.

It does not support:

- crawling or route discovery;
- authenticated sessions or stored credentials;
- remote scheduling or monitoring;
- vulnerability exploitation;
- framework compliance scoring;
- cloud synchronization, teams, or multi-tenancy.

## Workspace Invariants

1. Frontend and CLI use the same Python audit and assurance functions.
2. An imported target never runs without an explicit user action.
3. Raw response header values are not persisted in workspace summaries.
4. Query strings and fragments remain redacted by default.
5. Cross-origin redirects remain blocked by default.
6. The browser UI never receives or stores the server session token beyond
   page memory.
7. Framework mappings remain non-scoring and show limitations.
8. An approved baseline is never updated automatically.
9. A stale workspace revision never overwrites a newer revision.
10. The local server binds only to loopback in the baseline.
11. Import is bounded to 2 MiB, previewed, and explicitly confirmed before
    create or replacement; importing never starts an audit.
12. A workspace-only disabled-target list is separate from the v0.4 policy
    format. Disabled targets cannot run or enter a new candidate baseline.
13. Each completed audit records one bounded, data-minimized session summary;
    detailed response values remain in the active run and explicit exports.

## Primary Workspace Flow

1. Open or create a local workspace.
2. Add an authorized target with a stable label and explicit profile.
3. Set score threshold, required controls, reporting expectation, and
   cross-origin-isolation expectation.
4. Run an audit or complete assurance policy.
5. Inspect the profile decision, findings, evidence mappings, and limitations.
6. Export a detailed report.
7. Create a candidate baseline only from a successful run.
8. Review the diff and explicitly approve the baseline.
9. Re-run later to identify profile, score, and control-state regressions.

## Interface Surfaces

- **Targets:** authorized URLs, labels, profiles, and current run state.
- **Assessment:** current target evidence, score, and findings.
- **Assurance:** policy outcome, regressions, and baseline comparison.
- **History:** bounded local audit-session summaries, newest first.
- **Evidence:** framework relationships and primary sources for a selected
  control.
- **Workspace:** local storage status, import, export, clear data, and version
  information.

The first screen is the working target inventory and current assurance state,
not a landing page or marketing dashboard.

## Data Retention

Detailed current results remain in memory until replaced or exported. The
workspace persists only:

- policy configuration;
- approved baseline;
- latest summary per target;
- up to 50 data-minimized audit-session summaries;
- workspace versions and timestamps.

Persisted summaries exclude raw response headers and full error traces.

## Workspace Document Versioning

Workspace document schema `1.1` adds `disabled_target_ids`. Schema `1.2` adds
`audit_history`: at most 50 records containing a UUID audit ID, UTC completion
time, scope, policy name, outcome, exit code, and score-level target summaries.
Neither schema alters the v0.4 policy or methodology contract. Deterministic
migrations initialize both workspace-only fields as empty. Changing target
configuration or disabling a target clears its persisted latest summary; an
approved baseline and prior session history are retained deliberately so the
next eligible run can expose a reviewed change.

## Framework Interpretation

The evidence taxonomy is governed by
[`FRAMEWORK_ALIGNMENT_POLICY.md`](FRAMEWORK_ALIGNMENT_POLICY.md). ATT&CK and
D3FEND context never changes score, policy outcome, or regression state.

## Release Requirements

Version 0.5 is not complete until:

- loopback API authorization controls pass hostile-origin tests;
- persistence and migrations pass rollback and conflict tests;
- the UI completes the full target-to-assurance workflow;
- browser QA covers desktop and mobile;
- keyboard, focus, semantics, contrast, zoom, and reduced motion are reviewed;
- the frontend makes no third-party runtime requests;
- packaged wheel installation includes and serves the built UI;
- framework wording and mappings pass a primary-source review;
- every applicable item in the v0.5 release gate is evidenced.
