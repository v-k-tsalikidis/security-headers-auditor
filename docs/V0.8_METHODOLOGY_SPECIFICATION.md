# v0.8 Methodology And Delivery Specification

## Purpose

v0.8 adds a portable **review evidence capsule** for an already-completed
policy assurance or route assurance result. It solves a narrow review problem:
bind the exact evaluated scope, compact outcome, optional approved baseline,
and static profile definitions in one offline-verifiable artifact without
archiving a broad raw HTTP report.

The capsule is not a scanner, live-monitoring service, security attestation,
signature, compliance decision, vulnerability finding, or proof that a target
is secure.

## Inputs And Scope

Creation accepts exactly one of the following explicit scope sources:

1. a current policy JSON; or
2. a current same-origin route-comparison manifest.

It also requires the matching `--format review-json` output and may include the
approved baseline that that assessment actually used. Creation validates all
inputs before writing the output and does not make an HTTP request.

Policy capsule scope rejects query-bearing URLs, fragments, embedded credentials,
`include_query`, and cross-origin redirects. Route scope reuses the existing
bounded 2–25 same-origin route contract. A baseline must match the policy name
and target IDs, or the exact route manifest, respectively.

## Capsule Contract

The container is a deterministic uncompressed ZIP with fixed metadata and
sorted entry names. It contains only these regular files:

| Entry | Purpose | Data classification |
| --- | --- | --- |
| `scope.json` | Canonical effective policy or route-manifest scope. | Security review evidence: target host/path is necessary for review. |
| `assessment.json` | Compact, machine-readable score/control state and diagnostic codes. | Security review evidence. |
| `baseline.json` | Optional validated approved baseline. | Security review evidence. |
| `profile-definitions.json` | Current static profile and mapping definitions. | Product methodology evidence; contains no target or runtime data. |
| `capsule-manifest.json` | SHA-256 digest and byte size for every other entry. | Integrity metadata. |

The manifest schema is published at
[`docs/schemas/evidence-capsule-manifest.schema.json`](schemas/evidence-capsule-manifest.schema.json).
The implementation is stricter than the descriptive schema: it enforces the
exact entry set, sorted order, canonical JSON, and semantic bindings.

## Privacy And Claim Boundary

The review formats and capsule intentionally omit raw response-header values,
cookies, credentials, query strings, fragments, response metadata, target URLs
from the compact assessment, and diagnostic prose. Target scope remains only in
`scope.json`, because a reviewer cannot assess whether an approval applies
without the explicit authorized scope.

Operators must protect capsules as security review evidence. The manifest detects
content changes only when compared with a trusted expected digest; it does not
authenticate the author, approve a baseline, establish target ownership, or
prove the target's runtime behaviour.

## Offline Verification And Threat Model

`--verify-evidence-capsule PATH` performs no extraction, target request, or
third-party network call. It rejects:

- non-regular or symlink source files;
- over-limit source files, capsule files, and entries;
- duplicate, missing, additional, directory, encrypted, compressed, or
  non-deterministically timestamped ZIP entries;
- path traversal by accepting only the fixed entry-name allowlist;
- invalid UTF-8, duplicate JSON keys, non-finite numbers, non-canonical JSON,
  malformed schemas, incompatible methodology/mapping versions, and missing
  scope-to-assessment or scope-to-baseline bindings;
- SHA-256 size or digest mismatches.

The fixed upper bounds are 512 KiB per source/entry and 2 MiB for the complete
capsule. These limits are deliberate: a capsule is compact review evidence, not
a general archive format.

## Operator Workflow

Create the compact assessment from a known policy run, then package it without
performing a second audit:

```bash
security-headers-auditor --policy policy.json --baseline approved-baseline.json \
  --format review-json --output assurance-review.json

security-headers-auditor --create-evidence-capsule review.shac \
  --capsule-policy policy.json \
  --capsule-assessment assurance-review.json \
  --capsule-baseline approved-baseline.json

security-headers-auditor --verify-evidence-capsule review.shac
```

For routes, substitute `--route-comparison routes.json` and a route
`--format review-json` assessment, then use `--capsule-route-comparison` and an
optional `--capsule-baseline`. The assessment and baseline must be from the same
declared scope; invalid or mismatched inputs return exit code `2`.

## Versioning And Compatibility

The capsule manifest and compact review artifacts have independent schema
versions. Verification requires current methodology, evidence-mapping, policy
or route-manifest, baseline, and static profile-definition compatibility; it
never guesses how to migrate a future format. A changed methodology or mapping
set requires reviewed regeneration, not silent reinterpretation.

## Non-Goals

v0.8 does not add signing, key management, identity verification, hosted
storage, retention controls, target discovery, crawling, authentication,
fuzzing, browser automation, payload injection, vulnerability confirmation,
SIEM/ticketing integration, or automatic baseline approval. Those concerns need
separate authorization, controls, and a different product problem statement.
