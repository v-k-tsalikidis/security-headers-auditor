# v0.7 Controlled Route Assurance Specification

## Status

This document specifies the delivered v0.7.0 feature. The stable release and
its tag-triggered delivery evidence are recorded in the
[v0.7 release gate](RELEASE_GATE_V0.7.md). The tool version is `0.7.0`;
methodology remains `0.5.0` because v0.7 compares existing scored-control
states and does not change score, profile, CSP, policy, or evidence-mapping
semantics.

## Product Problem

An explicit route comparison exposes differences between known routes in one
run, but cannot reveal whether the same reviewed route set drifted between
releases. Manual report comparison tends to be unreliable and can spread raw
response-header evidence beyond the scope needed for change review.

## Contract

Route assurance adds a separate `1.0` baseline artifact. It contains:

- one canonical route-comparison manifest: name, exact origin, route IDs,
  query-free paths, and declared profiles;
- the methodology and evidence-mapping set versions;
- score and scored-control status, severity, points, and max-points per route;
- no response-header values, query strings, fragments, credentials, cookies,
  final response values, timestamps, or approval identity.

The baseline is generated only from a complete comparison run. Its writer does
not overwrite an existing path. The artifact is a candidate until the operator
reviews it and explicitly chooses to use it as an approved baseline.

The next run validates the artifact before comparison. Schema, artifact,
methodology, mapping-set, manifest, route ID, path, and declared-profile
differences are incompatible configuration errors. The implementation never
guesses an upgrade or automatically re-baselines changed scope.

## Drift Signals

Within the exact compatible scope, route assurance reports only:

- score decrease;
- worse scored-control status;
- worse scored-control severity;
- lower scored-control points;
- a newly actionable scored control; and
- a baseline scored control missing from the run.

In-run control variance remains a review signal. It does not fail the process
by itself. A route baseline is not a security policy, waiver, vulnerability
finding, compliance decision, coverage proof, or browser-runtime validation.

## Safety And Claims Boundary

- The mode requests exactly the operator-declared 2–25 routes on one origin.
- It does not crawl, discover paths, use queries, follow cross-origin redirects,
  authenticate, persist response headers, or schedule background work.
- Operational errors exit with code `2`; detected baseline regressions exit with
  code `1`; a complete run without regression exits with code `0`.
- Existing framework mappings remain supporting evidence only. v0.7 adds no
  new framework mapping, score, certification, compliance, endorsement, or
  security-effectiveness claim.

## Acceptance Requirements

v0.7 is complete only when deterministic tests prove that:

1. the baseline schema validates offline and rejects unknown/future fields;
2. a candidate is deterministic, data-minimized, and cannot be written from a
   partial run or overwrite an existing path;
3. schema, methodology, mapping, manifest, profile, and baseline tampering fail
   before a comparison result can be trusted;
4. score, status, severity, points, new-control, and missing-control regression
   signals are deterministic and distinguishable from operational errors;
5. the existing route comparison still requests exactly the declared route set;
6. CLI JSON and Markdown expose compact route assurance evidence without raw
   header values; and
7. the full regression suite, packaging, documentation, and release gate pass.
