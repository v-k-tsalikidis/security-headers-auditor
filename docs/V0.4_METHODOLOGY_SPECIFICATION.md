# v0.4 Methodology Specification

## Status

This document defines the normative v0.4 behavior of Security Headers Auditor.
The v0.3 response-profile scoring model remains intact. Version 0.4 adds a
separate continuous assurance layer so that policy, compliance-supporting
evidence, and baseline regressions do not distort the underlying response score.

## Design Invariants

1. A response score measures alignment with one declared response profile.
2. Assurance controls do not add points to or subtract points from that score.
3. Continuous assurance requires explicit endpoint profiles unless a policy
   deliberately enables conservative auto-detection.
4. An observed header is time-bound configuration evidence, not proof of
   application security, infrastructure security, or regulatory compliance.
5. A baseline is an approved comparison state, not a waiver or risk acceptance.
6. Tests and release verification do not depend on public remote targets.

## Continuous Assurance Policy

The policy schema is version `1.0` and is defined in
[`schemas/audit-policy.schema.json`](schemas/audit-policy.schema.json).

Every policy records:

- the methodology version it was written for;
- a human-readable policy name;
- shared defaults;
- one or more stable target identifiers;
- an authorized URL and explicit response profile for each target;
- score thresholds and allowable score decrease;
- severities that fail the policy;
- required controls;
- reporting-readiness and cross-origin-isolation expectations;
- redirect, query-retention, and timeout decisions.

Unknown fields are rejected. This prevents a misspelled setting from silently
weakening CI behavior. `auto` profiles are rejected in policy mode unless
`allow_auto_profile` is explicitly true.

## Assurance Expectations

Assurance controls use one of four expectations:

| Expectation | Meaning |
| --- | --- |
| `observe` | Report evidence without creating an absence failure |
| `recommended` | Treat missing or partial configuration as review work |
| `required` | Require a passing readiness result |
| `not_applicable` | Record an explicit policy decision not to assess the capability |

These expectations are policy inputs. They are not inferred from a header alone.

## Reporting Readiness Analysis

Reporting analysis evaluates two related controls:

### Reporting Endpoints

- parses `Reporting-Endpoints` as a constrained RFC 8941-style dictionary;
- requires quoted endpoint values;
- resolves relative references against the final response URL;
- rejects non-trustworthy endpoint URLs;
- rejects duplicate groups and malformed members;
- places a 16 KiB analysis limit on the observed field;
- recognizes `Report-To` only as a documented legacy compatibility mechanism.

### CSP Reporting Linkage

- inspects enforced and report-only CSP fields;
- resolves `report-to` group names against valid modern or legacy endpoint groups;
- identifies unlinked group references;
- reports `report-uri` as a legacy partial configuration;
- distinguishes configured reporting from verified report delivery.

A pass means that the observed response contains coherent configuration. It does
not prove browser delivery, collector availability, sanitization, retention,
triage, alerting, or incident response.

## Cross-Origin Isolation Bundle

The bundle evaluates response-level readiness rather than checking COOP, COEP,
and CORP as unrelated checkboxes.

A response-level pass requires:

- `Cross-Origin-Opener-Policy: same-origin`; and
- `Cross-Origin-Embedder-Policy: require-corp` or `credentialless`; and
- no invalid observed `Cross-Origin-Resource-Policy` value.

CORP is not required on the top-level response. Dependent resources must still
be validated through browser QA and their own CORS/CORP behavior. A pass does
not prove `window.crossOriginIsolated`, complete resource-graph compatibility,
or compatibility with payment, popup, and federated sign-in workflows.

## Versioned Compliance-Evidence Mappings

Mappings are stored in
`src/security_headers_auditor/data/compliance_evidence_v1.json`.

The manifest records:

- schema and mapping-set versions;
- last review date;
- framework name and version;
- requirement identifier;
- relationship type;
- evidence rationale;
- explicit limitations;
- source citation.

The fixed claims policy is `supporting-evidence-only`. Relationships such as
`supports`, `related`, and `legacy-support` must never be rendered as
`compliant`, `certified`, or `satisfies`.

## Approved Baselines

Baseline schema `1.0` records only comparison data:

- methodology and mapping-set versions;
- policy name;
- stable target identifier;
- redacted target;
- selected profile and score;
- finding status, severity, category, applicability, and points.

Raw header values are deliberately excluded. Baselines contain no runtime
timestamp, making equivalent snapshots deterministic.

A candidate baseline can be created only from an assurance run that has no
policy violations, regressions, or operational errors.

A baseline is incompatible when its schema, methodology, or mapping-set version
does not match the current tool. The operator must review the change and create
a new approved baseline.

## Regression Rules

Version 0.4 detects:

- a current target absent from the approved baseline;
- an approved baseline target absent from the current policy;
- a target URL change under a stable identifier;
- selected-profile change;
- score decrease beyond the target's declared tolerance;
- a control status moving to a worse state;
- a control severity or point contribution moving to a worse state;
- a new actionable control finding;
- a baseline control missing from a run under the same methodology version.

Status order is:

```text
pass/info/observed/not_applicable < review < warning < missing < error
```

Operational failures are not treated as security regressions. They use exit code
`2` because a reliable comparison could not be performed.

## CI Outputs

### SARIF 2.1.0

SARIF emits stable rule identifiers for policy violations, regressions, and
operational errors. Target identifiers and control keys are stored as result
properties. No source-code location is fabricated for a remote HTTP response.
Release artifacts are validated against the
[official OASIS SARIF 2.1.0 JSON Schema](https://docs.oasis-open.org/sarif/sarif/v2.1.0/os/schemas/sarif-schema-2.1.0.json).

### JUnit XML

JUnit emits one testcase per current policy target and adds a deterministic
synthetic testcase when a regression belongs only to a retired baseline target.
Policy violations and regressions become failures; audit execution problems
become errors. Failure text records the target, control, previous state, and
current state where those fields apply.

### Exit Codes

| Code | Meaning |
| ---: | --- |
| `0` | Assurance passed |
| `1` | Policy violation or regression |
| `2` | Configuration, compatibility, or operational error |

## Non-Goals

Version 0.4 does not:

- certify compliance;
- prove that reporting collectors receive or process reports;
- crawl resource dependency graphs;
- execute browser runtime validation;
- test DDoS protection, MFA, malware resistance, phishing defenses, or server hardening;
- replace application security testing, threat modeling, or authorized penetration testing;
- automatically accept or update a baseline.
