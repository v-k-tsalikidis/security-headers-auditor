# v0.6 Methodology And Delivery Specification

## Status

This document defines the in-development v0.6 scope. It is not a release
claim. The controlled CSP parser changes response-score semantics, so this
version uses methodology `0.5.0`. v0.4 policies and baselines are intentionally
incompatible and must be reviewed and re-baselined under the new methodology.

## Product Problem

Teams that maintain several application routes or CI policies need to know the
auditor's exact profile expectations without making an HTTP request, parsing a
human report, or reverse-engineering source code. They also need deeper,
bounded evidence from a route set and CSP syntax without confusing response
inspection with browser enforcement or compliance verification.

## v0.6 Scope

1. Deterministic, machine-readable profile-definition export for offline CI,
   review, and internal tooling.
2. Controlled multi-response assessment for an explicit, bounded route list,
   so operators can compare profile selection and findings per route without
   crawling or discovering endpoints.
3. A deeper CSP parser that identifies selected directive-level quality signals
   while preserving the boundary between static header inspection and full
   browser-policy validation.

## Profile-Definition Export Contract

The `--export-profile-definitions PATH` command produces canonical JSON with:

- profile IDs, labels, descriptions, applicability, weights, and rationale;
- rule purpose, recommendation, standards, and cited source identifiers;
- versioned supporting-evidence mappings with their relationship, confidence,
  rationale, and limitations;
- the referenced citation records and explicit claims policy;
- no timestamp, target URL, request, response, raw header, or audit result.

The command accepts no audit target or policy mode and performs no network I/O.
Stable ordering and canonical JSON serialization make equal source definitions
produce equal bytes. This makes source-control review and CI drift checks
possible without treating the export as an attestation about a live system.

The accompanying JSON Schema describes format `1.0`. Consumers must reject an
unknown schema version rather than infer altered semantics.

## Safety And Claims Boundary

- A profile is a configuration expectation, not a classification of a live
  endpoint. Runtime auto-detection remains deliberately separate.
- Framework mappings remain `supporting-evidence-only`; they never grant score
  and are not compliance, certification, endorsement, or effectiveness proof.
- Route comparison accepts only operator-provided routes, retains current
  authorization and redirect bounds, and never crawls.
- CSP parsing will not claim to execute browser behavior, inspect HTML `meta`
  policies or every response-header instance, resolve nonce lifecycle, validate
  third-party runtime behavior, or prove bypass resistance.

## Route-Comparison Contract

`--route-comparison PATH` accepts one versioned JSON manifest containing a
single exact origin and 2–25 explicit, query-free route paths. Each route must
have a stable ID and an explicit `app`, `api`, or `brochure` profile. The
manifest is an operator-supplied assessment scope, not a discovery seed.

For every listed route, the existing bounded read-only audit is invoked once.
The comparison groups successful results by the declared profile and exposes a
control variance only when two or more routes in that same group returned
different scored-control states. A variance is a review signal: different paths
can intentionally need different controls or values. It does not alter an
individual score, create a policy violation, or assert a vulnerability.

The comparison summary intentionally excludes raw response-header values. It
contains only the supplied route ID and path, redacted final URL, status,
profile, score, summary, scored-control state, and operational error when one
occurred. Detailed evidence remains available through the existing individual
audit outputs when the operator stores them in an authorized location.

The mode rejects paths that could change origin, all query strings, fragments,
credentials, duplicate route IDs, auto-selected profiles, and more than 25
routes. Cross-origin redirects remain blocked and cannot be enabled from this
manifest. It performs no crawling, route discovery, authentication, or
background scheduling. Route IDs and paths must each be unique.

## Acceptance Requirements

The profile export is complete only when:

1. every shipped profile and rule is represented exactly once in stable order;
2. each profile's scored weights remain 100;
3. output is deterministic and has no runtime timestamp or target data;
4. the exported manifest validates against the committed JSON Schema;
5. CLI tests prove no audit or network function is invoked;
6. incompatible audit/policy inputs are rejected rather than silently ignored;
7. documentation states the scope and limitations accurately.

The route comparison is complete only when:

1. its schema, origin, path, profile, count, and duplicate-ID bounds are
   validated before any request;
2. the run requests exactly the declared route set and preserves existing
   redirect behavior;
3. comparison occurs only within the same explicit profile and is deterministic
   for equal audit results;
4. output contains no raw header value, query, fragment, or global security or
   compliance verdict;
5. route variance and operational-error behavior have deterministic tests;
6. documentation makes the review-only scope and authorized-use boundary clear.

## CSP Parsing Contract

The v0.6 parser applies the W3C CSP Level 3 serialized-policy rules that are
relevant to this response-header tool: ASCII directive names are normalized,
source-value tokens retain their original case, the first duplicate directive is
effective, and later duplicates are recorded as ignored. It models a
comma-delimited policy list but deliberately records multi-policy enforcement as
review-needed rather than attempting a browser-equivalent policy intersection.

The CSP evaluator adds bounded signals for valid nonce/hash syntax, `data:` in
the effective script source list, duplicate directives, non-ASCII/invalid
directive tokens, and multi-policy ambiguity. It still evaluates only selected
header syntax and does not assert that a browser will enforce the policy as
intended. `unsafe-inline` with a valid nonce/hash remains a nuanced deployment
case, not evidence that nonce generation, reuse prevention, DOM injection
defenses, or browser compatibility are correct.

The CSP parser is complete only when:

1. duplicate, case, nonce/hash, `data:`, malformed-token, and policy-list
   semantics have deterministic fixtures;
2. the reporting-linkage parser shares the same first-duplicate behavior;
3. CSP score changes are documented and force methodology-aware re-baselining;
4. notes state the static-header and browser-validation limitations precisely;
5. prior profile, authorization, redirect, report-escaping, and framework
   evidence behavior remains regression-tested.

The final v0.6 release gate will add the acceptance evidence for the CSP-parser
work and complete release verification before any public tag or release.
